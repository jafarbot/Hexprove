## What We Removed

### ❌ Specific Company Names
Removed: CertiK, Hacken, Trail of Bits, OpenZeppelin, Cantina

**Why:** 
- We're not competing with them, so don't mention them
- Avoids creating comparisons or associations

### ❌ Recommendations
Removed: "we recommend working with..."

**Why:**
- Not our job to recommend security auditors
- Potential liability if they do poor work
- Keeps us focused on our own value prop

## Sales Talking Points (Updated)

### If Prospect Asks: "Do you do security audits?"

**Answer:**
"No, we focus on functional QA testing—the product experience layer. Security audits are a separate service category that focuses on smart contract code verification. Most crypto products need both: audited contracts AND thorough functional testing. We handle the functional testing side."

### If Prospect Asks: "Who should we use for security audits?"

**Answer:**
"That's a different service category from what we do. There are specialized security firms that focus on smart contract audits. Happy to discuss your functional QA needs, and you can explore security audit options separately."

**DON'T SAY:**
- ❌ "We recommend CertiK or Hacken"
- ❌ "Trail of Bits is good for that"
- ❌ "You should use OpenZeppelin"

**DO SAY:**
- ✅ "That's handled by specialized security firms"
- ✅ "There are firms that focus on smart contract audits"
- ✅ "That's a separate service category"

## Content Strategy Going Forward

### ✅ DO Write About:
- Functional bugs we catch (gas estimation, wallet connection, transaction flows)
- Testing DeFi protocols, wallets, NFT marketplaces
- Manual QA vs automation
- Crowdsourced QA problems
- Crypto-specific QA challenges

### ❌ DON'T Write About:
- Security audits vs QA (creates comparison)
- "When you need a security audit" (not our domain)
- Comparing ourselves to security firms
- Smart contract vulnerabilities (that's security audit territory)

### 🤔 Gray Area (Avoid Unless Necessary):
- "10 bugs that slip through security audits" (implies we're catching security issues)
- "After your security audit, you still need QA" (creates dependency narrative)

**Better approach:** Focus entirely on functional QA value without mentioning security at all.
